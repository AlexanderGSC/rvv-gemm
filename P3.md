### 2. Análisis de Ensamblador y Rendimiento

Si analizamos la salida de ensamblador vemos un detalle muy interesante: por defecto el compilador es muy conservador en la vectorización, y aplica un LMUL de 1. ### 4. 

La gráfica de rendimiento obtenida

![Resultado](P3sinsu.png)

Los resultados indican que el aumento en el LMUL no beneficia el rendimiento. Esto se debe a una combinación de factores:

* La arquitectura del chip SpaceMit K1 implementa un multiplicador vectorial de 8 lanes para un SEW de 32 bits, es decir, es capaz de procesar 8 multiplicaciones de float en paralelo. Sin embargo si vemos el código:

```C
for (int j=0; j < terms; ++j) {
      //Term = Term * v_x2
      v_term = VVMUL(v_term, v_x2, vl);
      // Tem = Term / C[i]
      v_term = VFMUL(v_term, C[j], vl);
      // sum = sum + Term
      v_sum  = VVADD(v_sum, v_term, vl);
    }
```

El bucle interno, que calcula los n términos de la serie de Taylor para un punto, tiene dos dependencias RAW encadenadas. Esto hace que el multiplicador procese 8 elementos de golpe, espere a que se vacíe el multiplicador, procese la siguiente multiplicación y por último procese la suma de los elementos. Aunque se utilice la instricción ```vmacc``` de multiplicación y suma acumulada no se ganaría gran cosa.
